Design System Architecture — Pick an Approach

How do we structure the token + component layer inside apps/mobile/src/?

A

Flat — single config file

All tokens (colors, spacing, typography) live in one tamagui.config.ts. Components in src/components/.

Pros

  • Simple to start
  • One file to find everything

Cons

  • Grows unwieldy fast
  • Hard to evolve color modes later
B

Layered — design/ folder ⭐ Recommended

Separate primitive tokens from semantic tokens, then assemble into Tamagui config.

src/
├── design/
│   ├── tokens.ts      ← primitive scales (all colors, size steps)
│   ├── themes.ts      ← semantic tokens (surface, text, etc.)
│   ├── typography.ts  ← type scale (h1–p, weights, sizes)
│   └── config.ts      ← assembles Tamagui config
├── components/
│   ├── Card.tsx
│   ├── Button.tsx
│   ├── Input.tsx
│   └── index.ts       ← barrel export
└── ...

Pros

  • Clear separation of concerns
  • Easy to add dark/light modes later
  • Scales well

Cons

  • More files than A
C

Monorepo package — packages/ui

Extract the entire design system into a shared monorepo package, ready for a future web app.

Pros

  • Shareable across apps
  • Clean boundary

Cons

  • Significant build config overhead
  • Premature — only one app today
  • Tamagui + monorepo requires careful setup
