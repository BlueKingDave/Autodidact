# ADR-008: Drizzle ORM for Database Access

## Status

Accepted — 2026-05-05

## Context

The application needs a database access layer for PostgreSQL (via Supabase). Requirements: TypeScript type safety, migration management, support for pgvector custom column types, and compatibility with the Supabase connection pooler. The ORM must handle complex queries (pgvector cosine similarity, JSONB columns for chat messages and course blueprints) while staying close enough to SQL to allow escape-hatch raw queries.

## Decision

We use Drizzle ORM (`@autodidact/db` package) for all database access across services.

## Consequences

### Positive

- SQL-close API: Drizzle queries look like SQL, making it easy to reason about query behavior and performance without needing to understand ORM query rewriting.
- Custom column types: the pgvector column type (`src/vector.ts`) integrates cleanly with Drizzle's `customType` system, handling `number[]` ↔ `[x,y,z,...]` serialization transparently.
- Lightweight: no runtime schema introspection; no "magic" query rewriting at runtime.
- TypeScript inference on query results without a code generation step (unlike Prisma's generated client), which simplifies the monorepo build graph.
- Works correctly with Supabase's transaction-mode pooler (port 6543), which is required in WSL2 environments where the direct PostgreSQL host is IPv6-only.

### Negative

- Drizzle's `.set()` method cannot handle the pgvector `::vector` cast for UPDATE statements — raw `db.execute(sql\`...\`)` is required for embedding updates, introducing an escape hatch that bypasses type safety.
- Less mature ecosystem than Prisma; fewer community tutorials and third-party integrations.
- No built-in seeding or data fixture tooling.

### Neutral

- Migrations are plain SQL files (not Drizzle push) — generated by drizzle-kit and reviewed manually before committing. This adds a review step but keeps the SQL explicit.

## Alternatives Considered

- **Prisma**: Rejected — Prisma's interactive transactions are incompatible with Supabase's transaction-mode pooler (port 6543), which is required in WSL2 environments (see the WSL2 note in `packages/db/CLAUDE.md`). Prisma also generates a client from the schema, requiring a build step that complicates the monorepo's turborepo task graph. pgvector support in Prisma requires raw query workarounds similar to what Drizzle requires anyway.
- **Raw pg / node-postgres**: Considered — maximum SQL control, but loses TypeScript inference on query results and requires a fully manual schema management approach. The complexity overhead was not justified given Drizzle's lightweight SQL-close model.
- **Kysely**: Considered — strong TypeScript support and SQL-close API comparable to Drizzle. Rejected because Drizzle's pgvector extension support and Supabase integration documentation were already being used in the project at the point of the decision, and switching would have required rewriting the custom vector type and the drizzle-kit migration tooling.
