# ADR-008: ORM / data access layer

## Status

Accepted
Date: 2026-05-10

## Context

`packages/db` is the single point of database access for the API, agent
(Postgres checkpointer in prod), and worker services. The data layer must:
type-check queries against the schema; generate and apply migrations;
serialize a custom column type (pgvector for course-topic embeddings, see
[ADR-010](./ADR-010-vector-search-strategy.md)); run on Cloud Run containers
with cold-start sensitivity; and stay legible to a solo team that reads the
generated SQL when something goes wrong.

This ADR builds on [ADR-002](../../cross-cutting/ADR-002-database-platform.md)
(Postgres on Supabase) and is consumed by every service that hits the DB.
The choice constrains migration tooling, query ergonomics, and how we add
custom types like `vector(1536)`.

## Non-goals

- Database platform — see [ADR-002](../../cross-cutting/ADR-002-database-platform.md).
- Vector indexing strategy — see [ADR-010](./ADR-010-vector-search-strategy.md).
- Schema design (which tables, which columns) — owned by `packages/db/src/schema/` and `data-model.md`.
- Connection pooling and Supavisor configuration — covered in `packages/db/CLAUDE.md`.

## Decision Drivers

- **TypeScript safety end-to-end** — query results must be typed against the schema. Untyped query results in 2026 are inexcusable.
- **Custom column types** — pgvector is a first-class need. Whatever ORM we pick must let us declare a `vector(N)` column and round-trip values without ceremony.
- **Migration workflow** — schema changes happen weekly; the ORM has to produce reviewable SQL we commit to the repo.
- **Cold-start cost** — Cloud Run cold-starts every container occasionally; bundle size and runtime initialization affect tail latency.
- **SQL transparency** — when a query is slow or wrong, we want to read the generated SQL and reason about it without re-implementing the ORM in our heads.
- **Learning curve** — solo team. The ORM should be conventional enough to onboard in an afternoon.
- **Future portability** — exit cost matters; ORMs with proprietary schema languages or runtimes are harder to leave.

## Options Considered

### Option A: Drizzle ORM (current)
**What it is:** TypeScript-first ORM. Schema declared in TS files (`pgTable(...)`); queries are a typed builder (`db.select().from(courses).where(eq(...))`); migrations are SQL files generated by `drizzle-kit` from schema diffs. Custom column types via `customType<...>()`.

**Pros**
- Schema in TypeScript — no separate schema language; the schema *is* TS, type inference flows directly.
- Custom column types are a primitive: our `vector()` type for pgvector lives in ~20 lines of `src/vector.ts`.
- Generates plain, readable SQL. We can paste any query into psql.
- Small footprint (~5 KB core gzipped) and minimal runtime initialization — kind to Cloud Run cold starts.
- Edge-runtime compatible (Cloudflare Workers / Vercel Edge), which we don't need today but keeps options open.
- Migrations are real `.sql` files that we review and commit; deploy artifact is auditable.
- `drizzle-kit studio` provides a usable DB inspector for ad-hoc debugging.

**Cons**
- Younger than Prisma; some patterns (joins with aggregations, complex CTEs) are more verbose to express in the builder.
- pgvector quirks: `db.execute(sql\`...\`)` is required for vector UPDATEs because the regular `.set()` path doesn't parameterize the `::vector` cast correctly. Documented in `packages/db/CLAUDE.md`, but the rough edge is real.
- "Relations API" (Drizzle's high-level join helper) is newer and less mature than Prisma's relations; we mostly write joins by hand.
- Smaller community than Prisma — fewer Stack Overflow hits, fewer plugins.
- Migration-generation occasionally produces noisy diffs that need manual cleanup before commit.

### Option B: Prisma
**What it is:** Schema-first ORM. Schema defined in `schema.prisma` (a DSL); generates a typed client; queries use a fluent builder. Prisma 7 (2025) replaced the Rust query engine with a TS/WASM compiler — addresses long-standing cold-start and bundle-size complaints.

**Pros**
- Mature, large community, ~3.8M weekly downloads (~2× Drizzle).
- Excellent DX for the common case: `prisma.user.findUnique({ where: { id } })` is hard to beat for ergonomics.
- Migration UX is highly polished: `prisma migrate dev` flows are well-trodden.
- Prisma Studio is best-in-class for a generated DB inspector.
- Prisma 7's TS/WASM engine reportedly gives ~3.4× faster queries on large result sets and ~9× faster cold starts vs the old Rust binary.

**Cons**
- Schema-first DSL (`schema.prisma`) is a separate language to learn and maintain. Cannot share types/utilities between the schema and the rest of TypeScript without code generation.
- Custom column types: pgvector is achievable via Prisma's `Unsupported("vector(1536)")` plus raw queries, but the experience is meaningfully worse than Drizzle. The pgvector + Prisma ecosystem in 2026 is still working through edge cases.
- Bundle size: even Prisma 7 ships a meaningful runtime (post-WASM-rewrite, it's still much heavier than Drizzle's ~5 KB).
- Generated client invocation is opaque — the SQL you actually run is hidden behind the client; you reach for `prisma.$queryRaw` or query logs to see it.
- Migration tooling has historically been "do not edit applied migrations"; if a generated migration is wrong, the un-rolling story is awkward.

### Option C: Kysely (pure query builder)
**What it is:** Type-safe SQL query builder, no schema definition language, no runtime generation. You define types matching your DB, Kysely uses them to type the builder. ~8 KB core.

**Pros**
- Tiny runtime, zero abstraction over SQL — the generated query is what you wrote, almost verbatim.
- Type safety at the same level as Drizzle for queries.
- Well-suited to "I want SQL but with autocomplete."
- Custom column handling is trivial because Kysely never claims to abstract the type system.

**Cons**
- Not actually an ORM — no schema DSL, no migrations, no relations helper. Migrations are typically handled by `kysely-codegen` + a separate migration runner (Kysely has a built-in migration system but it's bare-bones).
- We'd lose `drizzle-kit generate` and have to wire migrations manually, OR add another tool (e.g., `dbmate`, `node-pg-migrate`).
- Duplicated source of truth: types definition file + actual DB schema. If they drift, Kysely silently lies. Tools like `kysely-codegen` reduce this but it's another moving part.
- Smaller community (~550 K weekly downloads) than Drizzle or Prisma.
- For our case, we'd be doing more setup work to get back to Drizzle's level of ergonomics.

### Option D: Raw pg + zod schemas
**What it is:** Use `pg` (node-postgres) directly for queries; `zod` schemas (already used elsewhere — see [ADR-016](../schemas/ADR-016-runtime-schema-validation.md)) for runtime validation of query results.

**Pros**
- Zero abstraction, minimum dependencies.
- Maximum control over SQL.
- pgvector becomes "just a query" — no custom-type wiring needed.
- Compatible with anything; portable to any pg-speaking layer.

**Cons**
- We re-implement what every ORM gives for free: query result typing, migration tooling, parameter binding ergonomics, transaction helpers, schema reflection.
- Migration workflow becomes a bespoke tool choice (`dbmate`, `node-pg-migrate`, hand-rolled `psql` scripts).
- Type inference between SQL strings and result types relies on either manual zod schemas (decoupled, can drift) or a query-typing tool like `@ts-safeql/eslint-plugin` (good, but adds another moving part).
- Building blocks are great; assembled into an ORM is a project on its own.

### Option E: TypeORM
**What it is:** Mature, decorator-based ORM. Schema defined as TS classes with decorators; supports active record + data mapper patterns.

**Pros**
- Long history, large community, well-known.
- Decorators feel familiar to engineers from Java/Spring backgrounds.
- Supports many databases.

**Cons**
- Decorator-driven schema requires `experimentalDecorators` and `emitDecoratorMetadata` plus `reflect-metadata` — meaningful TS config impact.
- Active record pattern can hide query cost; common source of N+1 in production.
- Migrations have a long history of correctness issues (community reports of generated migrations producing wrong SQL).
- Custom column types are clunky.
- Project momentum slowed compared to 2018–2020; new TS-only projects in 2026 rarely pick it.

## Decision

**We use Drizzle ORM.**

## Rationale

Driver-by-driver:

- **TypeScript safety (#1)**: A, B, C, E all yes. D requires extra tooling.
- **Custom column types (#2)**: A wins decisively for pgvector. The `customType` API is documented and idiomatic; Prisma's `Unsupported` + raw query approach is meaningfully more friction. Kysely is good (no abstraction to fight).
- **Migration workflow (#3)**: A and B both excellent — Drizzle's plain `.sql` output we can review and edit suits us slightly better than Prisma's generated migrations because we frequently want to tweak before applying (e.g., the RLS policy migration).
- **Cold-start cost (#4)**: A best (~5 KB), C close (~8 KB), B reasonable post-Prisma 7 but still meaningfully heavier, D best (just `pg`), E worst.
- **SQL transparency (#5)**: A, C, D all transparent. B's runtime obscures generated SQL by default.
- **Learning curve (#6)**: A and B both modest. A wins on "no separate schema language to learn." C requires more upfront setup; we'd be reinventing migration tooling.
- **Future portability (#7)**: A has TypeScript schema files (no proprietary DSL); leaving Drizzle is "rewrite the query call sites" — non-trivial but contained. B's `schema.prisma` would have to be reverse-engineered into another schema format. C and D leave easily by construction. E's decorators couple to runtime metadata.

What we are sacrificing by picking Drizzle over Prisma: a slightly less polished Studio, a more verbose relations API, and the larger Prisma community when troubleshooting edge cases.

What we are sacrificing by picking Drizzle over Kysely: nothing material for our scale; Kysely would force us to assemble migration tooling separately for no net benefit on a Postgres-only stack.

What we are sacrificing by picking Drizzle over raw `pg`: convenience and a typed query builder. Worth it.

No reconsideration flag is raised. Drizzle is the first-principles choice
for our specific intersection (pgvector custom type, Cloud Run cold starts,
TypeScript-native team, migration files we want to review and commit).

## Consequences

### Positive
- Schema is a TypeScript module that the rest of the monorepo can `import`. Re-using column types or enum values across services is a trivial import.
- Migrations are plain SQL we can review, edit, and commit. The deploy artifact is auditable.
- pgvector custom type fits the ORM's primitives cleanly.
- Bundle size is kind to Cloud Run cold starts.
- Edge-runtime compatibility we may exploit later.

### Negative
- pgvector UPDATE quirk requires `db.execute(sql\`...\`)` with an explicit `::vector` cast — captured in `packages/db/CLAUDE.md`, but a real footgun.
- Relations API is less mature than Prisma's; complex multi-table joins are more verbose.
- Smaller community than Prisma means rarer issues take longer to find solutions for.
- Migration-generation noise sometimes needs manual cleanup before commit.

### Follow-up decisions
- Vector indexing and similarity search live in [ADR-010](./ADR-010-vector-search-strategy.md), which builds directly on Drizzle's custom column type.
- Reconsider this ADR if: relations API limitations start meaningfully slowing feature work (Prisma's relations would have shipped faster), Drizzle's project momentum stalls (e.g., maintainer / company changes), or we move to a database that Drizzle supports significantly worse than alternatives.
